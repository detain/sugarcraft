# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[env]** `unset GITHUB_TOKEN` before every `gh` invocation (`gh pr create`, `gh pr merge`, etc.) in this repo. The repo-level GITHUB_TOKEN clashes with the gh CLI's stored credentials and produces auth failures or wrong-account commits — used consistently across phase PRs to land cleanly.
- **[pattern]** When changes touch `candy-core`, regression-check the downstream packages before opening the PR: run `vendor/bin/phpunit` in `sugar-bits`, `sugar-prompt`, and `candy-shell` (chain them: `cd ... && vendor/bin/phpunit && cd ... && vendor/bin/phpunit`). They consume the runtime and break silently if a Cmd/Msg surface shifts.
- **[gotcha]** Don't bottom-stack a second `final class` inside an existing PSR-4 file (e.g. defining `Point` at the end of `src/Vector.php`). Composer autoloading expects one class per file matching the class name, so `use SugarCraft\Bounce\Point` fails with `Class X not found` even though tests in the same namespace see other classes fine — extract each class into its own file matching the class name.
- **[pattern]** For Bubble-Tea-style runtime ports, when adding a new sentinel Msg type that the Program intercepts (e.g. `SequenceMsg`, `ExecRequest`, `SuspendMsg`), the dispatch order matters: handle the sentinel branches in `Program::dispatch()` BEFORE the `WithFilter` pre-processor and BEFORE `model->update()`, otherwise filters can drop runtime sentinels and the Program will hang waiting for a teardown that never fires.
- **[pattern:click-counter-state-machine]** `ClickCounter` uses the `SugarCraft\Core\Concerns\Mutable` trait with a custom `mutate()` override: the trait provides `use Mutable;` declaration and the array-merge pattern (`new static(...array_merge(get_object_vars($this), $changes))`), but because `ClickCounter`'s mutable private fields (`clickCount`, `lastClickTime`, `lastClickZoneId`) are not constructor parameters, the override manually creates the instance and assigns private fields after construction. This hybrid approach keeps the mutable state separate from the readonly constructor config (`manager`, `clickIntervalMs`). See `candy-core/src/Concerns/Mutable.php` for the standard trait.
- **[arch:facade-over-candy-mouse]** CandyZone is a TEA-facing façade over `candy-mouse` (the SSOT low-level hit-test primitive). `Manager::mark()` → `SugarCraft\Mouse\Mark::wrap(prefix.id, content)`; `Manager::scan()` → `SugarCraft\Mouse\Scan::parse()` for bounds, then `stripMarkers()` removes the tags and returns the display-ready frame. `Zone\Zone` is a BC adapter: it keeps its public readonly bounds + `MouseMsg`-based `inBounds()`/`pos()`/`contains()`, but delegates the point-in-rect / width / height math to a wrapped `SugarCraft\Mouse\Zone` built from the same bounds (bridging `MouseMsg`→`Mouse\MouseEvent`). The marker WIRE FORMAT changed APC (`ESC _ candyzone:S:<id> ESC \`) → PUA sentinels (`U+E000 <id> U+E001 … U+E000 /<id> U+E001`) — behaviour (bounds, get/all/anyInBounds, DragTracker/HoverTracker/ClickCounter) is byte-identical because both scanners use the same close-marker union-rectangle math + candy-core `Width`.
- **[gotcha:id-charset]** `Mark::wrap()` (PR #1272) validates the marked id against `^[A-Za-z0-9._:-]+$` and throws `\InvalidArgumentException` otherwise, so `Manager::mark()` now throws on an out-of-charset `prefix().id` (only when enabled — disabled managers still pass content through unvalidated). Real consumer ids (`tab-N`, `cell:N`, `item:N`) and auto-prefixes (`1-`) already comply. Re-baseline any test asserting raw APC bytes to the PUA sentinels (`SugarCraft\Mouse\Sentinel::OPEN`/`CLOSE`); keep behavioural assertions as-is.
- **[gotcha:consumer-wire-format-test]** `sugar-bits` `TabsTest::testViewContainsZoneMarkersWhenManagerIsSet` asserts the OLD APC bytes (`\x1b_`, `candyzone:S:testtab-0`) in `Tabs::view()`. The APC→PUA façade change makes that assertion fail — it must be re-baselined to PUA sentinels inside sugar-bits (out of candy-zone's edit scope). The behavioural sugar-bits path (`testMsgZoneInBoundsActivatesMatchingTab`, mark→scan→`MsgZoneInBounds->zone->id`) stays green.
