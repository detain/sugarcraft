# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** A YAML comment-stripper that only handles `whitespace + #...` (`/\s+#.*$/`) leaves bare comment-only lines (`# top-level comment` with no leading whitespace) intact, and the `unparseable line` branch then throws on them. Use `/^\s*(?:#.*)?$/` for the "skip blank/comment" check so both indented and column-0 comment lines are dropped.
- **[gotcha]** `git add sugar-wishlist/` will sweep up vim swap files like `sugar-wishlist/.README.md.swp` if the editor was open while you staged. They show up in `git status --short` as `A  sugar-wishlist/.README.md.swp` — easy to miss in a long file list and end up committed. Either close the editor before staging, add `.*.swp` to a `.gitignore`, or stage explicit subdirs (`sugar-wishlist/src sugar-wishlist/tests sugar-wishlist/README.md ...`) instead of the whole directory.
- **[gotcha]** After `gh pr create` / `gh pr merge --delete-branch`, the script prints `Warning: N uncommitted changes` (any small N) even on a clean tree — this is `gh` complaining about the lockfile diff that `composer install` left behind, not a real uncommitted-source warning. Don't react to it; verify with `git status` if unsure.
- **[correction]** Adding a new sibling library to the monorepo requires updating THREE separate places, not two: root `composer.json` (require + repositories), `.github/workflows/ci.yml` matrix `lib:` list, AND `README.md` (library count, the table of libraries, and the test-loop shell snippet). Missing the CI matrix entry means the new lib's PHPUnit suite silently never runs in PRs — symptoms only surface much later when a regression slips through. **Why:** ci.yml is a hand-maintained list, not glob-driven. **How to apply:** after every `composer require sugarcraft/<lib>` registration, grep for the previous lib name in `.github/workflows/ci.yml` and `README.md` and add the new one alongside it.
- **[correction]** Phase 10 ("Polish & public launch — SugarBuzz") in `CONVERSION.md` is NOT complete just because the per-library code/tests are merged — it explicitly includes the `sugarbuzz.dev` website (`docs/` skeleton with palette + library grid + quickstart), VHS-recorded README GIFs, and a per-library README/composer.json polish pass. User flagged this directly: "did you do the sugarbuzz website and related tasks?" after the agent marked Phase 10 "done" in TodoWrite based only on library merges. **Why:** marking a multi-deliverable phase complete on partial work hides outstanding scope and creates a false signal of readiness. **How to apply:** before marking any phase "done" in TodoWrite, re-read that phase's section in `CONVERSION.md` end-to-end and check every `- [ ]` item — if presentation/distribution/website tasks exist, they belong to the phase too.
- **[gotcha]** `composer validate --strict` flags every `"sugarcraft/*": "@dev"` entry as `unbound version constraints (@dev) should be avoided` — this is EXPECTED in this monorepo and is the correct shape for path-repository sibling deps. Do NOT try to "fix" it by pinning to a version: there are no published tags yet, and the path repositories require `@dev`. Drop `--strict` or filter the unbound-`@dev` warnings when scripting validation across all libs.
- **[convention]** All per-library `composer.json` files share the same metadata block: `keywords` (array, lowercase, kebab-case + always include `"sugarcraft"` and the upstream Go project name like `"bubbletea"` / `"lipgloss"`), `homepage: "https://github.com/detain/SugarCraft"`, single author (`Joe Huss <detain@interserver.net>`, role `Maintainer`), and `support` block pointing to `issues` / `source` / `docs: .../blob/master/CONVERSION.md`. Insert these BEFORE the `require` block, AFTER `license`, to match the established ordering across all libs.
- **[pattern]** When auditing per-library files (READMEs, composer keys, etc.) across all libs, drive the loop from a hard-coded list. Current canonical 16: `for d in candy-core candy-sprinkles honey-bounce candy-zone sugar-bits sugar-charts sugar-prompt candy-shell candy-shine candy-kit candy-freeze sugar-glow sugar-spark candy-wish sugar-wishlist candy-metrics; do ...; done`. `wc -c < $d/README.md` flags missing/empty files immediately. Glob/find won't reveal an empty file is conceptually missing.
- **[pattern]** After registering a new lib in root `composer.json`, run `composer update sugarcraft/<lib> --quiet && composer show sugarcraft/<lib>` to verify it resolves through the path repository — `composer install` won't refresh the lock for a newly-added path package, only `composer update <name>` will, and `show` confirms the package metadata loaded correctly before pushing.

- **[gotcha]** `.github/workflows/vhs.yml` has a hand-maintained matrix `lib:` list (parallel to ci.yml). When adding `.vhs/*.tape` files to a library that isn't in the vhs.yml matrix, the workflow silently does NOT render any GIFs for it — `find . -name '*.gif' -path '*/.vhs/*' | wc -l` will show the count short by exactly the missing lib's tape count, but `gh run view <id> --json jobs` shows ALL listed jobs as `success` (no honey-bounce job exists in the matrix to fail). After adding tapes for a new lib, grep `.github/workflows/vhs.yml` for an existing lib name and add the new one alongside; then re-run `gh workflow run vhs.yml` and verify with `gh run view <id> --json jobs -q '.jobs[].name'` that `render (<new-lib>)` appears.
- **[pattern]** Public-site lib detail pages (`docs/lib/<slug>.html`) reference per-lib GIFs via `https://raw.githubusercontent.com/sugarcraft/<lib>/main/.vhs/<n>.gif` (the synced org repo's `main` branch), NOT a relative `.vhs/` path. The monorepo path is `<lib>/.vhs/<n>.gif` and that's where READMEs reference them. Don't accidentally use the monorepo-relative path in `docs/` — the GIFs won't load on the public site because GitHub Pages doesn't serve files outside the `docs/` directory.
- **[pattern]** When verifying CI render parity across all libs, audit with: `for lib in <canonical list>; do tapes=$(ls $lib/.vhs/*.tape 2>/dev/null | wc -l); gifs=$(ls $lib/.vhs/*.gif 2>/dev/null | wc -l); [ "$tapes" -gt 0 ] && [ "$gifs" -lt "$tapes" ] && printf "%s: %d tapes, %d gifs (missing %d)\n" "$lib" "$tapes" "$gifs" $((tapes-gifs)); done`. A non-zero "missing" delta with `gh run view ... -q .conclusion` returning `success` is the smoking gun for a vhs.yml matrix omission.

- **[pattern]** Before writing `examples/*.php` snippets that call a library's public API, verify the actual signature with `grep -n "public function" <lib>/src/<Class>.php` — don't trust memory of the constructor / fluent shape. Small APIs drift between iterations, and a wrong example ships compile errors to readers.
- **[gotcha]** A `for d in ...; cd /home/sites/SugarCraft/$d; vendor/bin/phpunit` loop without subshell parens (or `cd ... && cmd` chaining) leaks the working directory between iterations, and any `cd` that fails silently leaves you running the next command in the wrong place — output for libs after the first often comes back empty. Wrap each iteration body in `(cd "$d" && ...)` so the cwd resets per-loop.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/SugarCraft/sugar-charts && cmd`, the NEXT Bash call still starts in sugar-charts. Always either use absolute paths or prefix every Bash call with `cd /home/sites/SugarCraft && ...` to anchor it; if you get a mysterious empty response from `ls`/`cat`, suspect the cwd before suspecting the file.
- **[gotcha]** `Color::rgb($r, $g, $b)` throws `InvalidArgumentException: rgb component out of range [0,255]` (in `candy-core/src/Util/Color.php:39`) for any component > 255. When synthesising test pixel grids with arithmetic like `Color::rgb(64 * $c, 64 * $r, 128)`, wrap with `min(255, ...)` (or pick a step that can't exceed 255) when seeding pixel data in tests.
- **[gotcha]** Tests that assert env-var defaulting behavior (e.g. `Session::fromEnvironment()` falling back to `SSH_CLIENT` when `SSH_CONNECTION` is missing) must clear BOTH `$_SERVER` AND the process env. Clearing only `$_SERVER[$key]` leaves `getenv($key)` returning the host process's value (e.g. `USER=my` on the dev box). Pattern: snapshot `getenv($k)` for each key in `setUp()`, `unset($_SERVER[$k]); putenv($k);` to clear both, then restore both in `tearDown()`.

- **[gotcha]** `echo` and `exit` are PHP language constructs, NOT functions — so you cannot define `function echo(...)` (parse error) and you cannot call `\exit(main($argv))` (runtime `Call to undefined function exit()`). Both forms compile-/run-fail and cannot be "namespaced". When porting older code or using these in `match` arms (`'help' => echo(...)`), wrap the output in a real helper named `say()`/`out()` and use bare `exit(main($argv))` (no leading backslash) at the bootstrap line. Run `php -l <file>` on every CLI bin you touch — these failures point at the wrong line if you only spot-check at runtime.
- **[correction]** Monorepo-wide contributor docs (locale codes, naming policy, conversion roadmap, etc.) belong at the REPO ROOT alongside `MATCHUPS.md`/`CONVERSION.md`/`PROJECT_NAMES.md`, NOT under a specific lib's directory like `candy-core/lang/`. **Why:** every lib uses these codes when contributors add `lang/<locale>.php` files, and burying the doc inside `candy-core/` leaks contributor-only guidance into the downstream-distributed `sugarcraft/candy-core` Packagist package. **How to apply:** when creating a new cross-cut reference doc, place it at the root and add it to `AGENTS.md`'s "Source-of-truth files" list so future agents discover it.
- **[gotcha]** From a `.claude/worktrees/<name>/` worktree, `git checkout master` fails with `fatal: 'master' is already used by worktree at '/home/sites/sugarcraft'` because the primary worktree owns master. Don't try to switch — just `git fetch origin master --quiet && git reset --hard origin/master` to update the current detached HEAD, then `git checkout -b claude/<feature>` to start a fresh branch. Or work in a non-master branch from the start.
- **[pattern]** When patching `T::translate`'s lookup chain (e.g. adding base-language fallback `fr-fr` → `fr` → `en`), add three regression tests in lockstep: (1) regional locale falls back to base when no regional file exists, (2) regional file takes precedence over base when both exist, (3) regional locale falls all the way through to `en` when neither regional nor base file exists. Without all three, future refactors of the `??` chain can silently break ordering.
