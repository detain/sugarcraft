# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) ŌĆö do not edit manually.

- **[pattern]** Reverting a path-repo's `composer.lock` change that drifted during local work: `git checkout <consumer>/composer.lock` from the lib's own subdir works because git's pathspec is resolved relative to the repo root, not the CWD ŌĆö `cd candy-pty && git checkout ../candy-core/composer.lock` does NOT work (returns `pathspec did not match`). Either run from repo root (`cd /home/sites/sugarcraft && git checkout candy-core/composer.lock`) or use the absolute path. The lock drifts because `composer install` in a path-repo consumer rewrites the dep's `version` field to match the current branch ŌĆö leave it alone for PR work.
- **[gotcha]** In `candy-pty` pump tests that spawn a child to assert output bytes, prefer a single-write child like `printf 'TOKEN\n'` over multi-write `echo A; echo B` ŌĆö `echo` chains issue two separate `write()`s, and the pump's `STDIN_EOF_GRACE_SEC` deadline can trip between them, truncating the cassette / pump-output to just the first token. The flake reproduces ~1-in-3 even on a fast host. Single-write `printf` is race-free against the EOF grace window. Coverage of the multi-write path already lives in `candy-pty/tests/Posix/PosixPumpEofGraceTest.php`, which uses `sleep`-padded loops specifically to exercise that timing ŌĆö don't duplicate it from happy-path tests.
- **[gotcha]** When adding a new field to `candy-pty/src/PumpOptions.php` (or any immutable `with*` class where each mutator reconstructs via `new self(...all-fields-named...)`), you MUST update EVERY existing `with*` mutator to pass the new field through ŌĆö missing one silently drops the value when callers chain. PumpOptions currently has 8 mutators (withChunkBytes, withSelectTimeoutUs, withFlushDeadlineSec, withStdinEofGraceSec, withVEOF, withKeepalive, withOnSigwinch, withOnChildExit). Grep for the old field name in mutators (e.g. `grep -n 'onChildExit:' src/PumpOptions.php`) after the edit to verify count matches mutator count + 1 (constructor).
- **[pattern]** After adding a new cross-lib dev/runtime dep in the monorepo (e.g. `sugarcraft/candy-vcr` to candy-pty's `require-dev` + `repositories[]`), run `php tools/check-path-repos.php` from repo root to validate the full closure across all 46+ libs in <1s. Reports `closure clean` when every consumer's `repositories[]` contains a path-repo for every transitive `sugarcraft/*` dep, otherwise lists the missing entries per consumer. Faster and more comprehensive than manually inspecting each composer.json ŌĆö run this before committing any composer.json change that touches `repositories[]` or `require*`.
- **[gotcha]** `tools/check-path-repos.php` reports closure cleanliness based on actual filesystem path-repos (`{type:path, url:"../<dep>"}` entries), not just `require` keys ŌĆö so adding `"sugarcraft/candy-vcr": "@dev"` without a matching path-repo entry will pass `composer install` (via the parent `repositories[]` rollup at root) but fail check-path-repos. Both must be added to the lib's own `composer.json` for the closure check to stay green.
- **[gotcha]** `awalsh128/cache-apt-pkgs-action@v1` cannot reliably restore packages whose post-install scripts register `update-alternatives` symlinks (e.g. ffmpeg's transitive `libblas` dep, which creates `/usr/lib/x86_64-linux-gnu/libblas.so.3`). Even with `execute_install_scripts: true` the action's `dpkg --configure` invocation does NOT recreate update-alternatives entries ŌĆö ffmpeg still crashes at runtime with `libblas.so.3: cannot open shared object file`. Fix: skip the cache action entirely for packages that touch update-alternatives ŌĆö direct `sudo apt-get install -y --no-install-recommends <pkg>` on the runner-image's pre-indexed apt cache is ~15s and always runs real post-install scripts.

- **[gotcha]** `gh run list --workflow=ci.yml --branch=master --limit=1` returns the MOST RECENT run regardless of state ŌĆö a freshly-pushed merge commit shows up as `queued` with empty `conclusion` field, which is not "green CI" confirmation. To confirm CI is actually green, either bump `--limit=5` and visually find the most recent `completed\tsuccess` row, or filter via `gh run list --workflow=ci.yml --branch=master --status=success --limit=1`. The session bootstrap prompts that say "confirm master CI is green with `gh run list ... --limit=1`" are subtly wrong ŌĆö the call returns instantly but doesn't actually verify anything when a fresh merge is still in-flight.
- **[gotcha]** ALL `gh` subcommands (`gh api`, `gh pr create`, `gh run view`, `gh pr merge`, etc.) fail with `Bad credentials (HTTP 401)` when `GITHUB_TOKEN` is set in the environment to an invalid/expired token ŌĆö `gh` prefers the env var over its stored keyring auth and never falls back. Fix: `unset GITHUB_TOKEN && gh ...` (or `GITHUB_TOKEN= gh ...` as a one-shot prefix) once at the start of any session that uses `gh`. Already established for `gh pr create`; applies equally to read-only API calls like `gh run list --json status`.
- **[gotcha]** PHP fatal `Could not check compatibility between <FakeChildClass>::method(): <ConcreteStub> and <ParentInterface>::method(): <InterfaceType>, because class <ConcreteStub> is not available` fires when a test stub class declares a covariant return type pointing at ANOTHER class defined later in the same file. PHP's compatibility check runs at class-load time and cannot see classes that haven't been declared yet ŌĆö even though both end up in the same file. Fix: type stub method signatures against the INTERFACE (`public function master(): \SugarCraft\Pty\Contract\MasterPty`), not the concrete sibling stub class (`public function master(): FakeMasterPty`). Co-locating multiple stub classes in one test file is fine; just don't cross-reference them in return/parameter types ŌĆö reference the shared interface instead. Triggered while writing `tests/PtyPoolTest.php` where `FakePtyPair::master()` returned `FakeMasterPty` declared later in the same file.
- **[gotcha]** When writing a test stub `class Fake<X> implements <Interface>`, the method signatures must match the INTERFACE byte-for-byte including parameter names, default values, and nullable markers ŌĆö a stub for `MasterPty::read(int $len = 8192, ?float $timeout = null): ?string` written as `read(int $maxBytes, float $timeoutSec = 0.0): ?string` will pass syntax but fail at class-load with a `Declaration of ... must be compatible with ...` warning that PHPUnit's `failOnWarning=true` upgrades to a fatal. Same for `size(): array` ŌĆö the interface's `@return array{cols: int, rows: int, xpix: int, ypix: int}` PHPDoc is informational, but every consumer test that asserts on the array shape will break if the stub returns a 2-key subset. Always copy-paste the interface signature when writing a stub; don't paraphrase.

- **[gotcha]** PTY pump teardown race: when a fast child writes a small payload (e.g. `printf 'session-0-line\n'`) and exits before the pump's first `tick()`, naively marking the session done as soon as `$child->exited()` returns true causes those bytes to be lost ŌĆö they sit in the master buffer unread because the session was removed from the live set before the next `stream_select` round. Fix: watermark the moment the child is first observed exited (`$session->childExitedAt = microtime(true)`), then keep the session live until either master returns 0 bytes (EOF ŌĆö `read()` returns `''`) OR `now - childExitedAt > flushDeadlineSec` (e.g. 0.5ŌĆō1.0s). Reading master EOF immediately wins; the deadline is the cap. Canonical pattern: `candy-pty/src/Posix/MultiPump.php::tick()`. Same trap applies any time a PTY consumer checks `child->exited()` AND drains the master in the same loop ŌĆö always drain past the exit signal.
- **[pattern]** PtyPool / FFI-handle pool gating: when designing any "pool the expensive thing" wrapper around a candy-pty / candy-* primitive, measure the per-call cost FIRST before committing to a pool architecture. The libc cdef is cached as a per-process static in `Libc::lib()` (so handle reuse cost Ōēł 0.002ms/call, cold-load Ōēł 0.008ms/call, full `PosixPtySystem::open` Ōēł 0.05ms/call) ŌĆö no per-open FFI cost is left to amortize. PTY pairs are 1:1 with their slave path and can't be re-bound. The pool's value is then BOOKKEEPING (typed `PoolExhaustedException` at capacity, idempotent `release()`, `drain()` on shutdown) ŌĆö NOT handle reuse. Profile before scoping; if the cost is dust, scope the pool down to back-pressure + accounting only.
- **[gotcha]** Real-PTY tests in `candy-pty` MUST call `\stream_set_blocking($pair->master()->stream(), false)` before any read on the master ŌĆö without it, `fread` on the master fd ignores the `$timeout` argument and blocks the whole PHP process until the kernel buffer has data, defeating `Expect`'s deadline math and any `stream_select`-based loop. The slave inherits blocking-mode from the kernel's PTY default; only the master side needs the explicit flip. Symptom of forgetting: a test that should fail with `ExpectTimeoutException` instead hangs forever until the watchdog kills it.
- **[pattern]** When scripting a bash child in a PHP test, wrap the bash body in a quoted heredoc `<<<'SH' ... SH` (note the SINGLE-QUOTED delimiter). The single quotes block PHP variable interpolation so `$user`, `$$`, `$1`, etc. land literally in the bash child without escape-chain headaches. Combine with `IFS= read -r <var>` (not bare `read`) when the prompt protocol expects leading whitespace to be preserved ŌĆö bare `read` strips leading IFS chars and breaks `expect('login: ')`-style scripts that assume the line you typed is what bash receives. Canonical: `candy-pty/tests/ExpectTest.php::testScriptedLoginDialogAgainstRealBashChild`.
- **[gotcha]** PHP FFI's `\FFI::cdef()` eagerly resolves ALL declared symbols at library load time ŌĆö declaring a function the target libc doesn't export (e.g., POSIX 2024 `tcsetwinsize`/`tcgetwinsize` on glibc < 2.36, which is most non-bleeding-edge Linux distros, or macOS 15's libSystem.B.dylib despite the spec) causes the entire FFI handle to fail with `Failed resolving C function '<name>'`, breaking ALL subsequent FFI calls ŌĆö not just the missing one. Critically, gating the declaration behind `if (PHP_OS_FAMILY === 'Darwin')` does NOT help ŌĆö FFI still tries to resolve every symbol in the concatenated cdef block at load time. Fix: build the cdef block conditionally so the declaration is omitted entirely, not just guarded around the call site. Canonical pattern: `candy-pty/src/Libc.php::cdef()`.
- **[gotcha]** macOS arm64 + PHP FFI + variadic libc functions = ABI mismatch. Real libc `ioctl(int, unsigned long, ...)` is variadic, and the arm64 variadic ABI puts varargs on the stack while fixed args sit in `x0`ŌĆō`x7`. A fixed-arg PHP FFI cdef (`int ioctl(int fd, unsigned long request, void *arg)`) pushes the third arg into `x2` while the kernel reads it from the stack ŌåÆ ioctl returns `-1` with no useful errno. Workaround on Darwin: route through non-variadic POSIX 2024 wrappers (`tcsetwinsize`/`tcgetwinsize` for TIOCSWINSZ/TIOCGWINSZ) where available, or shell out to `stty -f /dev/fd/<fd>`. PHP FFI cannot reliably express variadic signatures ŌĆö declaring `int ioctl(int, unsigned long, ...)` breaks Linux too because FFI rejects `FFI\CData` arrays as variadic params ("FFI passing array is not implemented").

- **[gotcha]** Path-repo `composer.lock` mismatch on PR branches is EXPECTED, not a bug. `candy-core/composer.lock` (and any other consumer's lock) pins `sugarcraft/candy-pty` to `"version": "dev-master"`. When CI checks out a PR branch where the dep's HEAD is at `ai/<branch>`, composer reads the symlinked path-repo's branch as `dev-ai/<branch>` and rejects the lock as incompatible ŌĆö yielding a 7-cell failure pattern (Test PHP 8.3/8.4 ┬Ę PHPStan 8.3/8.4 ┬Ę macOS ┬Ę Windows ├Ś consumer-lib) on the PR. Do NOT regenerate the lock on the PR branch ŌĆö that just shifts the failure to the post-merge master push. Running `composer update sugarcraft/candy-pty` on a feature branch will likewise rewrite the consumer's lock to `"version": "dev-ai/<branch>"` ŌĆö if that lock gets staged, expect the same red CI; revert it (`git checkout <consumer>/composer.lock` from repo root) before committing.

- **[gotcha]** `candy-vcr` `Recorder::open($path)` WITHOUT an explicit header writes the JSONL header with the default 80├Ś24 cols/rows ŌĆö calling `$recorder->recordResize($cols, $rows)` AFTER `open()` lands a resize event but does NOT update the already-written header line. Fix: pass `Recorder::defaultHeader($cols, $rows, $runtimeId)` as the second arg to `Recorder::open()` whenever the recorded program runs at non-default dimensions. - **[pattern]** PTY-real PHPUnit tests skip cleanly on FFI-less / sandboxed CI via a 5-line `requirePtySyscalls()` preamble: `PHP_OS_FAMILY === 'Windows'` (windows skipped), `extension_loaded('ffi')`, `is_readable('/dev/ptmx') && is_writable('/dev/ptmx')`, `extension_loaded('pcntl')` (only when spawning with `controllingTerminal: true`), and `is_executable('/bin/echo')` (or whichever child binary you spawn). Mirrors `candy-pty/tests/`'s existing pattern ŌĆö copy verbatim when adding new lib tests that need a real PTY pair.
- **[pattern]** `vendor/bin/phpunit --display-deprecations` reveals the exact source (file:line) of `D`-marked deprecation warnings in test output, including which tests triggered each one. Use it when a `D` shows up after adding new tests ŌĆö it pinpoints whether the deprecation is in your new code or inherited from an existing dep (e.g. candy-pty's `PosixTermios.php:43` `FFI::new()` static-call deprecation). Without the flag the test runner just shows the count, not the source.
