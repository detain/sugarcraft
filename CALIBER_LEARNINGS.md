# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[fix]** `gh pr merge <n> --merge --delete-branch` returning `GraphQL: Pull Request is not mergeable (mergePullRequest)` is usually a transient mergeability-recompute race on GitHub's side — most often triggered by a fresh push that just landed (e.g. a Caliber follow-up commit pushed seconds before merge), or a CI-required-checks computation still in flight. Wait a few seconds (or run any other unrelated command to give GitHub time) then retry the same `gh pr merge` invocation. Verify success via `gh pr view <n> --json state,mergedAt`. Do NOT confuse this with actual merge conflicts — those report `Pull Request is not mergeable: the merge commit cannot be cleanly created` instead. The plain "not mergeable" message without elaboration almost always clears on retry.
- **[gotcha]** When testing signal-driven behavior (SIGWINCH forwarding, SIGCHLD reapers, EINTR retry), do NOT use a fixed `usleep()` to wait for the signal target to be "ready" before delivering — PHP startup + `proc_open` shim + exec setup is variable (~100-200ms locally, often longer in CI), and a fixed delay races with that. Symptom: the test asserts `assertContains(100, $cols)` for the pre-signal baseline but observes only post-signal values like `[150, 150, 150, …]` because by the time the inner program's first sample landed, the signal had already been delivered and processed. Fix pattern: poll an observable side-effect (a tempfile being appended-to, a stream becoming readable) until you see the BASELINE state, deliver the signal, then poll again until you see the POST-SIGNAL state. Both polls bound by a generous deadline (~5s) so a hung supervisor doesn't time out the whole suite. Avoid the fixed-`usleep`-then-`posix_kill` shape entirely.
- **[gotcha]** Backgrounding a debug supervisor in a Bash one-liner via `(timeout N cmd &)` (subshell-wrapped + timeout-wrapped) BREAKS signal delivery — the captured `$!` ends up being the subshell pid or the timeout wrapper pid, not the inner PHP/cmd, so `kill -WINCH $!` either no-ops or kills the wrapper instead of the supervisor. The supervisor either exits before the loop body runs (no observable resize) or never receives the signal at all. Fix: drop the subshell and timeout — use direct `cmd & SUP_PID=$!; sleep 0.3; kill -WINCH $SUP_PID; wait $SUP_PID 2>/dev/null` so `$!` is the actual supervisor pid. If you need a timeout, build it into the inner cmd itself (`for i in 1..20; do … sleep 0.05; done` self-bounds at ~1s) rather than wrapping with `timeout(1)`.
- **[pattern]** For middleware that needs a transport-injected dependency (e.g. `Spawn` needs the PTY supervisor's `runChild`), use a duck-typed `setTransport()` injection seam: the transport's `run()` walks `$stack` and calls `$mw->setTransport($this)` on any middleware where `method_exists($mw, 'setTransport')` is true. The middleware itself doesn't need to implement an interface — just declaring the method is enough. This keeps the middleware contract clean (no transport coupling on basic Logger / Auth) while letting opt-in middleware grab transport-specific capability. The same seam doubles as a transport-incompatibility detector: a middleware that's only valid under `HostSshdTransport` (e.g. `BubbleTea` — runs the Program inline using supervisor stdio, would collide with InProcess's bytes pump) defines `setTransport()` to set an internal `$inProcessMode = true` flag, then `handle()` checks the flag and throws a migration error. HostSshd doesn't perform the injection, so the legacy code path runs unchanged. Test it with a probe `new class implements Middleware { public function setTransport(ChildSpawner $t) {…} public function handle(…) {} }` that captures the injected instance.
- **[pattern]** VHS CI workflow auto-discovers tapes within a registered lib — `.github/workflows/vhs.yml` renders EVERY `.vhs/*.tape` in the lib via `for tape in .vhs/*.tape; do vhs "$tape"; done` (around line 182), gated by `hashFiles(format('{0}/.vhs/*.tape', matrix.lib)) != ''`. Adding a new tape to an already-registered lib needs NO workflow changes — just commit the new `.tape`. The hand-maintained bash-array gates which libs are eligible; the per-lib tape set is glob-driven from there.
- **[gotcha]** VHS `.tape` files require ALL `Env KEY value` values to be quoted strings — `Env COLUMNS 100` and even `Env USER demo` error with `COLUMNS expects string` / `USER expects string` from the VHS parser. Always quote: `Env COLUMNS "100"`, `Env USER "demo"`, `Env TERM "xterm-256color"`. Numeric-looking values and plain identifier-looking values BOTH need quotes — the parser only accepts the quoted-string form. Same convention applies to other directives that take value args.
- **[env]** Local `vhs <tape>` rendering on Ubuntu hosts without snap fails with `Command '/usr/bin/chromium-browser' requires the chromium snap to be installed` — VHS uses headless chromium under the hood and the apt package isn't sufficient. Local verification path: run `vhs <tape>` to surface PARSER errors (these fire before the chromium launch), fix syntax, then push and let CI render the GIF (CI image has the right chromium). Do not waste time trying to install the snap inside agent-driven sessions — the parser-only check catches every tape-author error worth fixing locally.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/sugarcraft/<lib> && cmd`, the NEXT Bash call still starts in `<lib>` — `git status` will then show paths with `../` prefixes that look like they're outside the repo but aren't. A bare `ls candy-core/src/` from inside `candy-core` returns NO output (empty `{}` response from the tool, not a `No such file or directory` error) because the path is relative to the wrong cwd. A bare `vendor/bin/phpunit` after the cwd has drifted to the repo root produces `timeout: failed to run command 'vendor/bin/phpunit': No such file or directory` because phpunit is per-lib (in `<lib>/vendor/bin/`), not at the repo root. Bare `grep candy-core/src/Cmd.php` after `cd .../candy-vcr` produces `ugrep: warning: candy-core/src/Cmd.php: No such file or directory`. Worse, a compound command like `git status --short && git add foo bar baz && git status --short` can return an EMPTY `{}` response (no stdout, no stderr, no exit-code error) when the cwd has drifted to a worktree state where the chained command finds nothing — and the same shape (`git status && unset GITHUB_TOKEN && gh pr merge <n> ...`) returns empty `{}` when the merge fails because of an uncommitted file blocking the merge readiness check. Always anchor with `cd /home/sites/sugarcraft && ...` or `cd /home/sites/sugarcraft/<lib> && ...` to start from the right place. Same trap bites HEREDOC-written `/tmp/dbg.php` scripts: `require_once __DIR__ . '/vendor/autoload.php'` resolves `__DIR__` to `/tmp/`, NOT to the working dir of the heredoc-emitting Bash call. Use absolute autoload paths (`require_once '/home/sites/sugarcraft/<lib>/vendor/autoload.php';`) in any standalone PHP script you stage under `/tmp/`.
- **[gotcha]** `gh pr create` prints `Warning: 1 uncommitted change` to stderr but STILL creates the PR successfully and returns 0 — the warning is informational, not a failure. The PR will not contain that uncommitted file (it stays staged on the branch). Recovery: after `gh pr create` returns the URL, check `git status` for staged-but-not-committed files; if any belong with the change-set, follow up with a separate commit + push BEFORE attempting `gh pr merge` — an uncommitted file on the branch can also cause `gh pr merge` to no-op with an empty response while GitHub's mergeability check is computing. Do NOT abort the PR or `git stash` blindly — the warning is benign for ancillary files. The most common cause: Caliber's pre-commit-hook refresh touches `<lib>/CALIBER_LEARNINGS.md` (often just normalising the title from `# Caliber Learnings — <lib>` to `# Caliber Learnings`) AFTER your `git add` set was finalised. Inspect the diff before deciding whether to fold it into a follow-up commit or leave it for the next session.
- **[gotcha]** When asked to "fix failing tests" for specific libs, run `vendor/bin/phpunit` for each lib FIRST before assuming work is needed — the libs may already be green locally and the actual failure is on CI (e.g. PHP 8.1 matrix runner) due to language-version drift. The local-vs-CI gap is the most common false positive: candy-vt / candy-vcr / candy-wish / candy-metrics all reported `OK` locally on PHP 8.3, but the user's real complaint was the GitHub Actions PHP 8.1 leg failing. Don't go fishing for code bugs when the fix is bumping `require.php` to `^8.2` and dropping the 8.1 matrix entry in `.github/workflows/ci.yml`.
- **[convention]** When a lib uses a PHP 8.2+ language feature (readonly classes, DNF types, `true`/`false`/`null` standalone types, `new` in initialisers etc.) it MUST bump `composer.json` `require.php` to `^8.2` AND drop its row from the PHP 8.1 leg of `.github/workflows/ci.yml`'s matrix. The monorepo's baseline stays at `^8.1` for libs that don't need it; only opt out per-lib when a real syntax/runtime requirement forces it. Update the lib's README badge/install snippet too if it cites a PHP version.
- **[gotcha]** Composer prints `Warning: The lock file is not up to date with the latest changes in composer.json. You may be getting outdated dependencies. It is recommended that you run \`composer update\` or \`composer update <package name>\`.` to STDERR but still installs from the existing lock and tests still pass. The warning is a real signal — somebody touched `composer.json` (often a sibling-dep version bump or a new path-repo entry) without re-running `composer update`. Recovery: run `composer update sugarcraft/<dep>` (or full `composer update` if the change touches multiple) and commit the lockfile delta. Don't leave the warning lingering — CI on a clean checkout will install something different from what local devs see.
- **[gotcha]** When firing `task(subagent_type="explore", run_in_background=true)` agents to look up information you can answer with a single Grep/Glob/Read, you're paying the multi-LLM round-trip cost (and burning extra-usage budget) for trivia. Locating `candy-wish/composer.json` and reading its `require.php` is a one-line `Read /home/sites/sugarcraft/candy-wish/composer.json` — not a delegated explore-agent task. Reserve sub-agents for genuinely multi-step exploration ("find every consumer of X across N libs"), not single-file lookups.

- **[convention]** When shipping a v1 milestone PR for a SugarCraft lib, the 256-square candy-themed icon (`media/<slug>.png` + `docs/img/icons/<slug>.png`) referenced from `docs/lib/<slug>.html` and the README badge area CANNOT be generated by the agent — image creation is outside the tool surface. Call out the missing icon explicitly in the PR body under "Plan deviations" so reviewers don't treat it as an accidental omission, and so the follow-up icon-only commit is unblocked. Don't fake it with a stock placeholder — leaving the link broken is a clearer signal that art is owed.
- **[pattern]** When splitting incremental work across multiple PRs (e.g. PR1 introduces an abstraction, PR2-PR4 land the real implementation), keep the PR1 stub functionally equivalent to existing behaviour. Add a doc comment explicitly marking the stub state (`**PR1 stub.** The full implementation lands in PR2-PR4.`) so reviewers don't mistake stand-in code for finished work, and add tests that pin the PR1 contract so the stub doesn't silently regress while the real implementation is in flight.
- **[gotcha]** Public classes in this monorepo are `final` by project convention ("unless extension is part of the contract"). PHPUnit tests CANNOT mock or wrap them via an anonymous `new class($args) extends FooMiddleware { … }` — PHP fatals with `Class SugarCraft\…\Foo@anonymous cannot extend final class SugarCraft\…\Foo`. Use a sibling `implements` of the same interface as a probe/observer instead. This swap also dodges the temptation to weaken the production type by removing `final` purely for testability.

- **[correction]** When building dependency-aware CI gating (the `scripts/affected-libs.php` graph), `require-dev` deps are explicitly EXCLUDED from the edge set — only `require[sugarcraft/*]` counts as a real edge. **Why:** a phpunit/test-helper bump or other dev-only churn in candy-core should not fan out and retest every downstream lib; tests already exercise the runtime contract, not the dev-time toolchain. **How to apply:** when extending the affected-libs graph (new edge sources, new force-all triggers, new matrix shapes), keep `require-dev` out of `build_forward_graph()` and any future graph builders. If you need to track a dev-only relationship, do it in a separate map, never the main affected-libs graph.
- **[correction]** Dynamic affected-libs CI gating applies to `ci.yml` ONLY (tests / coverage / phpstan / windows / macos) — do NOT fan it out into `vhs.yml`. **Why:** vhs.yml already has its own changed-path scoping (`'*/.vhs/*.tape'`, `'*/examples/**'`, `'*/src/**'`, `'*/bin/**'`) that skips renders for libs whose visual surface didn't move; layering reverse-dep fan-out on top would re-render unrelated GIFs whenever an upstream dep changes — wasteful and non-deterministic. **How to apply:** keep the affected-libs.php script free of any vhs-matrix output; route every CI gating decision through ci.yml's `build-matrix` job and let vhs.yml's existing `paths:` trigger + per-lib bash array gate stand on its own.
- **[pattern]** `scripts/affected-libs.php` is the source of truth for which libs run in CI — it discovers libs by globbing `*/composer.json` AND requiring `*/phpunit.xml` (the marker for "testable lib"), then walks `require[sugarcraft/*]` reverse-deps. Adding a new lib with both files means ci.yml matrices auto-include it; NO hand-edit of ci.yml's matrix arrays needed anymore. Force-all triggers (entire monorepo retested): root `composer.json`/`composer.lock`, `.github/workflows/ci.yml`, or `scripts/affected-libs.php` itself in the change-set. workflow_dispatch and unresolvable git ranges (shallow clone, first commit on branch) also force-all from the workflow side.
- **[convention]** Six libs in the monorepo have ZERO `require[sugarcraft/*]` deps — `candy-hermit`, `sugar-calendar`, `sugar-readline`, `sugar-stickers`, `sugar-table`, `sugar-toast`. A `candy-core` change does NOT fan out to them, and they only appear in the affected-libs matrix when their own files change or a force-all trigger fires. Don't be surprised when a downstream-of-everything refactor's CI run skips these six — it's correct.
- **[gotcha]** When validating a `scripts/affected-libs.php`-style PHP CLI tool that writes JSON to stdout and debug to stderr, test it across these scenarios at minimum: `--all` (forces all), a `candy-core` file (max fan-out), a leaf-only lib file (`sugar-crush/src/Game.php` → 1 affected), a mid-graph lib (`candy-sprinkles`), a force-all trigger (`composer.json`, `.github/workflows/ci.yml`), and a no-op file (`README.md` → empty matrix). The empty-matrix case is the trickiest: the JSON must emit an empty array, not omit the key — GitHub Actions's `strategy.matrix.include` of `[]` spawns zero cells but a missing key fatals the workflow.
- **[gotcha]** Verifying `scripts/affected-libs.php` scoped behaviour with `--range HEAD~1..HEAD` is unreliable — if the last commit touched any force-all trigger (`.github/workflows/ci.yml`, root `composer.json`/`composer.lock`, or `scripts/affected-libs.php` itself) the script reports `force_all=yes, changed=0, affected=46/46` and you NEVER exercise the reverse-dep walk. Symptom: `force_all: true` in the JSON output even though you expected a single-lib slice. Fix: pick a range whose diff excludes those trigger files — `HEAD~2..HEAD~1` against an older candy-core-only commit gives the canonical scoped result (`force_all=no, changed=1, affected=40/46`). Always inspect `git diff --name-only <range>` before drawing conclusions about scoping correctness.
- **[pattern]** Quick CI-config-touch lint combo before committing changes to `scripts/affected-libs.php` + `.github/workflows/ci.yml`: `php -l scripts/affected-libs.php && python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))" && echo ci.yml: OK` — covers PHP syntax errors AND ci.yml YAML well-formedness in one shot. Catches both regressions cheaply before push; YAML parse errors in a workflow file silently fail the workflow on GitHub with no useful surface error.
